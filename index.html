Embedded Systems & STM32 Development Documentation

This comprehensive guide covers the theoretical foundations of Real-Time Systems, RTOS, Timers, and Serial Communication, and bridges the gap to practical implementation with six detailed STM32 microcontroller projects using C and the HAL library.

1. Real-Time Operating Systems (RTOS)

A Real-Time Operating System (RTOS) is a specialized operating system designed specifically to run applications with highly precise timing and a high degree of reliability. The hallmark of an RTOS is Determinism—the guarantee that a system will respond to an event within a strict, predictable timeframe.

RTOS vs. GPOS: A General Purpose Operating System (GPOS) like Windows, Linux, or Android prioritizes throughput and human interaction. It can tolerate occasional lag. An RTOS (like FreeRTOS, VxWorks, or Zephyr) prioritizes deadlines. A late answer in an RTOS is treated as a wrong answer.

Super Loop vs. RTOS

Before RTOS, embedded systems relied heavily on the "Super Loop" (Bare-metal) architecture.

  • Super Loop (while(1)): The application runs in a single, infinite loop. Tasks are executed sequentially. Interrupts handle synchronous I/O. It is highly efficient for small microcontrollers (typically < 64KB code size) but scales poorly. If one task gets stuck or takes too long, the whole system freezes, missing real-time deadlines.
  • RTOS Architecture: The application is divided into multiple independent tasks. A scheduler rapidly switches the CPU context between these tasks based on priority, creating the illusion of parallel processing (pseudo-parallelism).

Task States in an RTOS

In an RTOS environment (specifically focusing on FreeRTOS paradigms), a task can exist in one of four primary states:

  1. Running: The task is currently executing and occupying the CPU. In a single-core MCU, only one task can be in this state at any exact moment.
  2. Ready: The task is not blocked or suspended and is ready to execute, but a task of equal or higher priority is currently in the Running state.
  3. Blocked: The task is waiting for a specific event (e.g., a time delay vTaskDelay, waiting for data on a queue, or waiting for a semaphore). While blocked, it consumes zero CPU cycles.
  4. Suspended: The task has been explicitly paused (e.g., using vTaskSuspend()). It will remain in this state indefinitely until explicitly resumed.

Inter-Task Communication & Synchronization

Using global variables to share data between tasks is dangerous. It leads to race conditions and data corruption (e.g., one task reading half a variable while another task is writing to it). An RTOS provides kernel objects to handle this safely.

1. Queues

Queues are the primary form of inter-task communication. They operate as First-In-First-Out (FIFO) buffers.

  • Atomicity: Reading and writing to a queue are protected by the kernel, making them immune to interrupts and context switches during the operation.
  • Copy by Value: FreeRTOS copies the actual data into the queue. This means the sending task can immediately reuse the original variable safely.
  • Blocking: A task can be configured to block (wait) for a defined timeout period if it tries to read from an empty queue or write to a full queue.

2. Mutexes (Mutual Exclusion)

A Mutex is a "key" used to protect a shared resource (like a UART port or a global array). A task must "take" the mutex before accessing the resource and "give" it back when done.

  • Ownership: A mutex implies strict ownership. The task that takes the mutex must be the one to give it back.
  • Priority Inheritance: Mutexes temporarily raise the priority of the task holding the mutex to match the priority of any higher-priority task waiting for it. This prevents a critical bug known as Priority Inversion.

3. Semaphores

Semaphores are signaling mechanisms. Unlike mutexes, they do not imply ownership. Task A can take a semaphore, and Task B can give it.

  • Binary Semaphores: Can count to a maximum of 1. Ideal for Deferred Interrupt Processing (an ISR gives the semaphore, unblocking a high-priority task to do the heavy lifting).
  • Counting Semaphores: Used to track multiple identical resources or count the number of events that have occurred.

2. Real-Time Systems

A Real-Time System is an information processing system that must respond to externally generated input stimuli within a finite and specified period. The correctness of the system depends not only on the logical output but on the time at which the output is produced.

Types of Real-Time Constraints

  • Hard Real-Time: Meeting every deadline is absolutely critical. A missed deadline causes catastrophic system failure, potential loss of life, or massive financial loss. Examples: Pacemakers, Anti-lock Braking Systems (ABS), Air Traffic Control, Nuclear Reactor Control.
  • Firm Real-Time: Missing a deadline makes the computed result completely useless, but the system won't instantly fail, and lives aren't directly at risk. The missed result is simply discarded. Examples: Stock market trading systems, autonomous agricultural weeders.
  • Soft Real-Time: Deadlines are important for performance, but missing them simply results in degraded service quality rather than total failure. Examples: Video streaming, online gaming, ATM machines.

CPU Scheduling Algorithms

In multi-tasking environments, an algorithm must decide which task runs next. Tasks are defined by their Release time ($R_i$), Execution Time ($C_i$), Deadline ($D_i$), and Period ($P_i$).

Algorithm Type Mechanism & Priority Rule
FCFS (First Come First Served) Non-Preemptive Processes run in the exact order they arrive. Highly unpredictable response times.
SJF (Shortest Job First) Non-Preemptive The CPU is allocated to the task with the smallest execution burst time. Minimizes average wait time but can starve longer tasks.
SRTF (Shortest Remaining Time First) Preemptive The preemptive version of SJF. If a new task arrives with a shorter execution time than what is remaining for the current task, a context switch occurs.
RR (Round Robin) Preemptive Every task gets an equal, fixed "Time Quantum" (e.g., 10ms). Tasks cycle continuously. Excellent for fairness but bad for strict real-time deadlines.
RM (Rate Monotonic) Static / Offline Priority is assigned based on the task's period. The shorter the period, the higher the priority. Optimal for independent periodic tasks where Deadline = Period.
DM (Deadline Monotonic) Static / Offline Priority is assigned based on the task's deadline. The shorter the deadline, the higher the priority.
EDF (Earliest Deadline First) Dynamic / Online The scheduler constantly evaluates absolute deadlines. The task whose deadline is closest to the current time gets the highest priority.
LLF (Least Laxity First) Dynamic / Online Priority goes to the task with the smallest Laxity. Laxity = Deadline - Current Time - Remaining Execution Time. It represents the "slack" time a task has left.

3. Hardware Timers

A Hardware Timer is a dedicated silicon peripheral within a microcontroller designed specifically to track time, generate pulses, or measure external signals, entirely independently of the CPU core. This frees the CPU to process data instead of running wasteful software delay loops.

The Time-Base Unit

The core of any general-purpose timer consists of three crucial 16-bit or 32-bit registers:

  1. Prescaler (PSC): The raw system clock (e.g., 72MHz) is usually too fast. The prescaler divides this clock frequency down.
    Timer Clock = System Clock / (PSC + 1)
  2. Counter (CNT): This register increments (or decrements) by 1 every time the prescaler outputs a clock pulse (a "tick").
  3. Auto-Reload Register (ARR): This represents the "ceiling" of the timer. When the CNT reaches the value stored in the ARR, an Update Event (UEV) is generated, an interrupt can be triggered, and the CNT resets to 0.
Frequency Calculation: Update Frequency = System Clock / ((PSC + 1) * (ARR + 1))

Timer Interaction & Hardware Channels

Advanced timers (like TIM1, TIM2, TIM3 on STM32) have independent "Channels" capable of specialized functions:

  • Output Compare: The timer forces a pin High or Low when the CNT matches a specific Capture/Compare Register (CCR) value. Used for precise event triggering.
  • Input Capture: When a hardware pin detects an edge (rising or falling), the current value of the CNT is instantly copied into the CCR. Used for measuring the frequency or pulse width of external signals.
  • PWM Generation (Pulse Width Modulation): Rapidly switching a pin High and Low. The ARR dictates the Frequency, and the CCR dictates the Duty Cycle (the percentage of time the signal is High).
    Duty Cycle % = (CCR / (ARR + 1)) * 100
  • One-Pulse Mode (OPM): Generates exactly one pulse of a precise length in response to a hardware trigger, then halts.

The SysTick Timer

Unlike General-Purpose timers which are vendor-specific (STM32 vs NXP vs TI), the SysTick timer is universally embedded deeply within the ARM Cortex-M architecture itself. It is a 24-bit down-counter dedicated to providing the fundamental "heartbeat" (Tick) for Real-Time Operating Systems. It uses registers like SYST_RVR (Reload Value) and SYST_CVR (Current Value).

4. Serial Communication

Serial communication transmits data sequentially over a single wire or pair of wires (one bit at a time). It drastically reduces the pin count and wiring complexity compared to Parallel communication.

UART (Universal Asynchronous Receiver/Transmitter)

UART is Asynchronous, meaning no clock signal is shared between the two devices. Because there is no clock, both sides must agree on a precise transmission speed beforehand, known as the Baud Rate.

  • Wiring: 2 Wires: TX (Transmit) and RX (Receive), plus a shared Ground.
  • Data Frame: Idle state is HIGH. Transmission begins with a Start Bit (pulling line LOW). Followed by 5 to 9 Data Bits (sent LSB first). An optional Parity Bit (Even or Odd) is used for basic error checking. Finally, 1 or 2 Stop Bits (pulling line HIGH) return the bus to idle.
  • Pros & Cons: Simple, requires only two pins, allows full-duplex. However, speeds are relatively slow, strictly limited to two devices, and highly susceptible to timing drift if baud rates mismatch.

SPI (Serial Peripheral Interface)

SPI is a Synchronous, high-speed protocol heavily used for short-distance board-level communication (SD cards, LCD displays, fast sensors).

  • Wiring: 4 Wires: SCLK (Clock), MOSI (Master Out Slave In), MISO (Master In Slave Out), SS/CS (Slave Select).
  • Architecture: Master-Slave. The Master generates the clock. Communication is full-duplex (data is sent and received simultaneously in a shift-register ring topology).
  • Clock Configuration: Devices must align on Clock Polarity (CPOL) and Clock Phase (CPHA), yielding 4 possible SPI modes (Mode 0, 1, 2, 3).
  • Pros & Cons: Extremely fast (often >10 Mbps), continuous data streaming without start/stop bits. However, it requires 4 wires, and adding more slaves requires adding more individual SS pins, causing PCB routing complexity.

I2C (Inter-Integrated Circuit)

I2C combines the multi-device capability of SPI with the low pin count of UART. It is a Synchronous protocol.

  • Wiring: 2 Wires: SDA (Serial Data) and SCL (Serial Clock). Both lines require external pull-up resistors.
  • Architecture: Supports Multi-Master and Multi-Slave configurations. Slaves are targeted using unique 7-bit or 10-bit hardware addresses.
  • Protocol Flow:
    1. Start Condition: Master pulls SDA low while SCL remains high.
    2. Address Frame: Master sends the 7-bit Slave Address + 1 bit for Read/Write indication.
    3. ACK/NACK: The targeted slave pulls SDA low to Acknowledge receipt.
    4. Data Frames: Data is transmitted 8 bits at a time (MSB first), followed by an ACK.
    5. Stop Condition: Master lets SDA float high while SCL is high.
  • Pros & Cons: Highly scalable (up to 127 devices on two wires), robust built-in addressing and acknowledgment. However, it operates as Half-Duplex and is generally slower than SPI (Standard Mode: 100 kHz, Fast Mode: 400 kHz).

5. STM32 Practical Projects (C / HAL Library)

The following six projects demonstrate how to implement the theoretical concepts discussed above using STMicroelectronics' HAL (Hardware Abstraction Layer) in C. These are typically generated inside main.c using STM32CubeIDE.

Project 1: Bare-Metal GPIO & SysTick Blinky

The "Hello World" of embedded systems. We will toggle an LED using the HAL implementation of the ARM SysTick timer to generate a precise 1-millisecond delay.

/* Project 1: Basic LED Blinky using SysTick Delay */
#include "main.h"

void SystemClock_Config(void);
static void MX_GPIO_Init(void);

int main(void)
{
    /* Reset of all peripherals, Initializes the Flash interface and the Systick. */
    HAL_Init();

    /* Configure the system clock */
    SystemClock_Config();

    /* Initialize all configured peripherals */
    MX_GPIO_Init();

    /* Infinite loop (Super Loop Architecture) */
    while (1)
    {
        /* Toggle LED connected to GPIO Port A, Pin 5 (Standard Nucleo Green LED) */
        HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);
        
        /* Delay for 500 milliseconds. 
           HAL_Delay() relies directly on the ARM Cortex SysTick timer.
           The SysTick is configured inside HAL_Init() to generate an interrupt every 1ms. */
        HAL_Delay(500);
    }
}

static void MX_GPIO_Init(void)
{
    GPIO_InitTypeDef GPIO_InitStruct = {0};

    /* Enable GPIOA clock */
    __HAL_RCC_GPIOA_CLK_ENABLE();

    /* Configure GPIO pin Output Level */
    HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5, GPIO_PIN_RESET);

    /* Configure GPIO pin : PA5 */
    GPIO_InitStruct.Pin = GPIO_PIN_5;
    GPIO_InitStruct.Mode = GPIO_MODE_OUTPUT_PP; // Push-Pull mode
    GPIO_InitStruct.Pull = GPIO_NOPULL;
    GPIO_InitStruct.Speed = GPIO_SPEED_FREQ_LOW;
    HAL_GPIO_Init(GPIOA, &GPIO_InitStruct);
}

Project 2: Hardware Timer Interrupts (Non-Blocking Delay)

Using HAL_Delay() traps the CPU in a loop. Here, we use General Purpose Timer 3 (TIM3) to generate an interrupt every 1 second, freeing the while(1) loop for other tasks.

/* Project 2: Timer Interrupts. Assuming a 72MHz System Clock */
#include "main.h"

TIM_HandleTypeDef htim3;

void SystemClock_Config(void);
static void MX_GPIO_Init(void);
static void MX_TIM3_Init(void);

int main(void)
{
    HAL_Init();
    SystemClock_Config();
    MX_GPIO_Init();
    MX_TIM3_Init();

    /* Start the Timer Base generation in Interrupt Mode */
    HAL_TIM_Base_Start_IT(&htim3);

    while (1)
    {
        /* CPU is free to do other tasks here! No blocking delays. */
    }
}

static void MX_TIM3_Init(void)
{
    /* TIM3 Clock = 72 MHz. 
       Goal: 1 Hz (1 second) interrupt.
       Calculation: 72,000,000 / (PSC+1) / (ARR+1) = 1
       Let PSC = 7199 (Clock drops to 10 KHz)
       Let ARR = 9999 (10 KHz / 10000 = 1 Hz) */
       
    htim3.Instance = TIM3;
    htim3.Init.Prescaler = 7199;
    htim3.Init.CounterMode = TIM_COUNTERMODE_UP;
    htim3.Init.Period = 9999;
    htim3.Init.ClockDivision = TIM_CLOCKDIVISION_DIV1;
    htim3.Init.AutoReloadPreload = TIM_AUTORELOAD_PRELOAD_ENABLE;
    HAL_TIM_Base_Init(&htim3);
}

/* ISR Callback: Automatically called by HAL when TIM3 reaches ARR */
void HAL_TIM_PeriodElapsedCallback(TIM_HandleTypeDef *htim)
{
    if (htim->Instance == TIM3) {
        /* Toggle LED exactly every 1 second */
        HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);
    }
}

Project 3: PWM Generation (Fading an LED)

Instead of just toggling on and off, we use Timer Output Compare functionality to generate a PWM signal on TIM3 Channel 1. By adjusting the Capture/Compare Register (CCR), we change the duty cycle and dim the LED.

/* Project 3: PWM Generation for LED Fading */
#include "main.h"

TIM_HandleTypeDef htim3;

int main(void)
{
    HAL_Init();
    SystemClock_Config();
    MX_GPIO_Init();
    MX_TIM3_Init();

    /* Start PWM signals on TIM3 Channel 1 */
    HAL_TIM_PWM_Start(&htim3, TIM_CHANNEL_1);

    uint16_t dutyCycle = 0;

    while (1)
    {
        /* Gradually increase brightness */
        for(dutyCycle = 0; dutyCycle < 1000; dutyCycle += 10)
        {
            /* Dynamically update the CCR1 register */
            __HAL_TIM_SET_COMPARE(&htim3, TIM_CHANNEL_1, dutyCycle);
            HAL_Delay(10);
        }
        /* Gradually decrease brightness */
        for(dutyCycle = 1000; dutyCycle > 0; dutyCycle -= 10)
        {
            __HAL_TIM_SET_COMPARE(&htim3, TIM_CHANNEL_1, dutyCycle);
            HAL_Delay(10);
        }
    }
}

static void MX_TIM3_Init(void)
{
    TIM_OC_InitTypeDef sConfigOC = {0};

    htim3.Instance = TIM3;
    htim3.Init.Prescaler = 71;    // 72MHz / 72 = 1MHz timer clock
    htim3.Init.Period = 999;      // 1MHz / 1000 = 1kHz PWM frequency
    HAL_TIM_PWM_Init(&htim3);

    sConfigOC.OCMode = TIM_OCMODE_PWM1;
    sConfigOC.Pulse = 0;          // Initial duty cycle (0%)
    sConfigOC.OCPolarity = TIM_OCPOLARITY_HIGH;
    sConfigOC.OCFastMode = TIM_OCFAST_DISABLE;
    
    /* Attach the OC config to Channel 1 */
    HAL_TIM_PWM_ConfigChannel(&htim3, &sConfigOC, TIM_CHANNEL_1);
}

Project 4: UART Communication (Transmit & Receive Interrupts)

Configuring Asynchronous Serial Communication. We send a startup message via polling, then set up a non-blocking interrupt mechanism to echo received characters back to a PC terminal.

/* Project 4: UART Polling Transmit & Interrupt Receive */
#include "main.h"
#include <string.h>

UART_HandleTypeDef huart2;

/* Global buffer for receiving 1 byte */
uint8_t rx_buffer[1];

int main(void)
{
    HAL_Init();
    SystemClock_Config();
    MX_USART2_UART_Init();

    char *msg = "System Booting... \r\nEnter a character:\r\n";
    
    /* Transmit via Polling (Blocking). Wait max 100ms. */
    HAL_UART_Transmit(&huart2, (uint8_t*)msg, strlen(msg), 100);

    /* Arm the UART Receive Interrupt. 
       It will trigger when exactly 1 byte is received. */
    HAL_UART_Receive_IT(&huart2, rx_buffer, 1);

    while (1)
    {
        // Main loop is free
    }
}

static void MX_USART2_UART_Init(void)
{
    huart2.Instance = USART2;
    huart2.Init.BaudRate = 115200;
    huart2.Init.WordLength = UART_WORDLENGTH_8B;
    huart2.Init.StopBits = UART_STOPBITS_1;
    huart2.Init.Parity = UART_PARITY_NONE;
    huart2.Init.Mode = UART_MODE_TX_RX;
    huart2.Init.HwFlowCtl = UART_HWCONTROL_NONE;
    huart2.Init.OverSampling = UART_OVERSAMPLING_16;
    HAL_UART_Init(&huart2);
}

/* ISR Callback: Automatically called when the requested bytes are received */
void HAL_UART_RxCpltCallback(UART_HandleTypeDef *huart)
{
    if(huart->Instance == USART2)
    {
        /* Echo the character back to the terminal */
        HAL_UART_Transmit(&huart2, rx_buffer, 1, 10);
        
        /* Re-arm the interrupt for the next character */
        HAL_UART_Receive_IT(&huart2, rx_buffer, 1);
    }
}

Project 5: SPI Master (Reading a Sensor Register)

Implementing Synchronous full-duplex communication using SPI. We manually toggle the Chip Select (CS) pin to initiate communication, send a register address, and receive the response data concurrently.

/* Project 5: SPI Master Communication */
#include "main.h"

SPI_HandleTypeDef hspi1;

int main(void)
{
    HAL_Init();
    SystemClock_Config();
    MX_GPIO_Init();
    MX_SPI1_Init();

    uint8_t txData[2] = {0x8F, 0x00}; // 0x8F: Example read command (MSB=1 for read)
    uint8_t rxData[2] = {0, 0};
    
    while (1)
    {
        /* Step 1: Pull Chip Select (CS) LOW to activate slave */
        HAL_GPIO_WritePin(GPIOA, GPIO_PIN_4, GPIO_PIN_RESET);
        
        /* Step 2: Transmit and Receive simultaneously. 
           Send address, get junk. Send junk, get data. */
        HAL_SPI_TransmitReceive(&hspi1, txData, rxData, 2, 100);
        
        /* Step 3: Pull Chip Select (CS) HIGH to deactivate slave */
        HAL_GPIO_WritePin(GPIOA, GPIO_PIN_4, GPIO_PIN_SET);
        
        /* rxData[1] now contains the sensor data */
        HAL_Delay(1000);
    }
}

static void MX_SPI1_Init(void)
{
    hspi1.Instance = SPI1;
    hspi1.Init.Mode = SPI_MODE_MASTER;
    hspi1.Init.Direction = SPI_DIRECTION_2LINES;
    hspi1.Init.DataSize = SPI_DATASIZE_8BIT;
    hspi1.Init.CLKPolarity = SPI_POLARITY_LOW;   // CPOL = 0
    hspi1.Init.CLKPhase = SPI_PHASE_1EDGE;       // CPHA = 0 (Mode 0)
    hspi1.Init.NSS = SPI_NSS_SOFT;               // Manual CS control via GPIO
    hspi1.Init.BaudRatePrescaler = SPI_BAUDRATEPRESCALER_16;
    hspi1.Init.FirstBit = SPI_FIRSTBIT_MSB;
    HAL_SPI_Init(&hspi1);
}

Project 6: FreeRTOS Task Management & Queues

Moving away from the Super Loop, this project implements a Real-Time Operating System structure. We define two tasks and an inter-task communication queue. Task 1 produces data and sends it to the queue. Task 2 consumes the data and acts on it.

/* Project 6: FreeRTOS Integration with CMSIS-OS wrapper */
#include "main.h"
#include "cmsis_os.h"

/* RTOS Handles */
osThreadId_t ProducerTaskHandle;
osThreadId_t ConsumerTaskHandle;
osMessageQueueId_t DataQueueHandle;

/* Task Prototypes */
void StartProducerTask(void *argument);
void StartConsumerTask(void *argument);

int main(void)
{
    HAL_Init();
    SystemClock_Config();
    MX_GPIO_Init();

    /* Init scheduler */
    osKernelInitialize();

    /* Create Queue: holds up to 5 unsigned 32-bit integers */
    DataQueueHandle = osMessageQueueCreate(5, sizeof(uint32_t), NULL);

    /* Create Tasks */
    const osThreadAttr_t Producer_attributes = {
        .name = "Producer",
        .priority = (osPriority_t) osPriorityNormal,
        .stack_size = 128 * 4
    };
    ProducerTaskHandle = osThreadNew(StartProducerTask, NULL, &Producer_attributes);

    const osThreadAttr_t Consumer_attributes = {
        .name = "Consumer",
        .priority = (osPriority_t) osPriorityAboveNormal,
        .stack_size = 128 * 4
    };
    ConsumerTaskHandle = osThreadNew(StartConsumerTask, NULL, &Consumer_attributes);

    /* Start the FreeRTOS scheduler */
    osKernelStart();

    /* We should never get here as control is now taken by the scheduler */
    while (1) {}
}

/* Task 1: Produces data and writes to the queue */
void StartProducerTask(void *argument)
{
    uint32_t count = 0;
    for(;;)
    {
        count++;
        /* Send count to queue. Timeout = 0 (do not block if full) */
        osMessageQueuePut(DataQueueHandle, &count, 0, 0);
        
        /* Block task for 500 OS Ticks (usually 500ms). 
           This yields the CPU to other tasks. */
        osDelay(500);
    }
}

/* Task 2: Waits for data in the queue */
void StartConsumerTask(void *argument)
{
    uint32_t receivedData;
    for(;;)
    {
        /* Wait indefinitely (osWaitForever) until data arrives in the queue.
           Task is in BLOCKED state consuming 0 CPU cycles until data arrives. */
        osMessageQueueGet(DataQueueHandle, &receivedData, NULL, osWaitForever);
        
        if (receivedData % 2 == 0) {
            /* Toggle LED on even numbers */
            HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);
        }
    }
}
